05

URL Shortener

Base62 · CQRS · Cache-Aside · Bloom Filter

PostgreSQL Redis Spring Boot 3.3 Java 21 Hexagonal Architecture

Given a long URL, produce a short alias. When visited, redirect to the original.

Read traffic dominates write traffic 100:1. Every design decision flows from this asymmetry.

Why This Is Hard

Scale constraints

  • Redirect latency must be < 10ms → cache is mandatory
  • 100:1 read/write ratio → read path must be optimised separately
  • Viral links → millions of hits on one short code

Correctness constraints

  • Short codes must be globally unique
  • Codes must be URL-safe (no + or /)
  • Cache-busting attacks must be blocked cheaply

What we will build

  • POST /api/v1/shorten → returns short code
  • GET /r/{code} → 302 redirect to original

Patterns used

  • Base62 encoding — no-collision ID → code
  • CQRS — separate write and read paths
  • Cache-Aside — Redis in front of PostgreSQL
  • Bloom Filter — blocks random code enumeration

Algorithm: Base62 Encoding

Alphabet: 0-9 a-z A-Z — 62 characters, all URL-safe.

Why not Base64? Base64 adds + and / — special characters in URLs. Require percent-encoding, making short codes ugly and fragile.

Why auto-increment ID, not hash?

Hash URLAuto-increment ID
CollisionPossibleNever
Extra DB checkEvery writeNone
ComplexityO(1) + retry loopO(1) always
ALPHABET = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ"

encode(id):
  while id > 0:
    result.prepend(ALPHABET[id % 62])
    id = id / 62

decode(code):
  for each char in code:
    result = result * 62 + ALPHABET.indexOf(char)

Examples:
  ID = 1      → "1"
  ID = 62     → "10"
  ID = 125000 → "W0s"
  62⁶ = 56 billion URLs in 6 chars max

Key insight: shortCode is never stored in DB. Derived from ID at runtime in both directions.

Data Structure: Bloom Filter

Probabilistic bit array. Answers: "does this code exist?"

Properties

  • No false negatives — NO means definitely absent → safe 404
  • ~1% false positives — YES means maybe present → check cache/DB

Why it matters

Without Bloom Filter: random code enumeration forces a DB query per miss. With it: definite misses absorbed at O(k) in memory — never reach Redis or PostgreSQL.

Implementation

Redis SETBIT / GETBIT — 3 hash functions, 10M bit array ≈ 1.2MB. Persistent across restarts. Shared across instances.

Read path with Bloom Filter

GET /r/{code}
↓
Bloom Filter: mightContain(code)?
NO (definite miss)
404 — no DB hit
YES (maybe)
Redis cache check
↓
302 Redirect

Pattern: CQRS

Two completely separate paths. Write side persists. Read side queries. Never mixed.

Write Side — Command

POST /api/v1/shorten
↓
ShortenUrlUseCase
↓
INSERT into PostgreSQL → get ID
↓
encode ID → Base62 code
↓
add code to Bloom Filter
↓
return ShortCode

Read Side — Query

GET /r/{code}
↓
ResolveUrlUseCase
↓
Bloom Filter check
↓
Redis Cache-Aside
↓
PostgreSQL (on miss)
↓
302 redirect

Pattern: Cache-Aside (Lazy Loading)

Application manages cache explicitly. Cache sits beside the DB, not in front of it transparently.

Cache HIT path

  • Redis GET url:{code} → value found
  • Return original URL immediately
  • No DB query — latency < 1ms

Cache MISS path

  • Redis returns null
  • decode code → id via Base62
  • Query PostgreSQL by primary key
  • Store result in Redis with 24h TTL
  • Return original URL

Why 302 not 301?

301 is cached permanently by browsers. If we ever update or delete a link, the old redirect is stuck in every browser cache forever. 302 re-checks every time.

// ResolveUrlUseCase — Cache-Aside logic
public String resolve(String code) {

  // Bloom Filter gate
  if (!bloomFilter.mightContain(code)) {
    throw new UrlNotFoundException(code);
  }

  String cacheKey = "url:" + code;

  // Cache-Aside: check cache first
  return cache.get(cacheKey)
    .orElseGet(() -> {

      // Cache MISS: decode → DB lookup
      long id = encoder.decode(code);
      Url url = urlRepository.findById(id)
        .orElseThrow(() ->
          new UrlNotFoundException(code));

      // Populate cache for next request
      cache.put(cacheKey, url.originalUrl());
      return url.originalUrl();
    });
}

Hexagonal Architecture — Layers

Dependencies always point inward. Domain knows nothing about Spring, Redis, or JPA.

domain/ — pure Java, zero Spring
  • Url, ShortCode — value objects
  • Base62Encoder — domain service
  • UrlRepository, CachePort, BloomFilterPort — ports (interfaces)
  • UrlNotFoundException — domain error
application/ — use cases
  • ShortenUrlUseCase — write side
  • ResolveUrlUseCase — read side
  • No @Service, no @Autowired — wired by AppConfig
infrastructure/ — adapters
  • JpaUrlRepository — implements UrlRepository
  • RedisCacheAdapter — implements CachePort
  • RedisBloomFilterAdapter — implements BloomFilterPort
  • AppConfig — Spring wiring
api/ — controllers + DTOs
  • UrlShortenerController — @RestController
  • ShortenRequest / ShortenResponse — DTOs
  • GlobalExceptionHandler — 404 mapping

MVC → Hexagonal mapping

MVC nameHexagonal name
@RestControllerController in api/
@ServiceUseCase in application/
@Repository (interface)Port in domain/
@Repository (impl)Adapter in infrastructure/
@EntityJPA entity in infrastructure/

Why UseCase not Service?

One class, one action. ShortenUrlUseCase has one method. No fat service with 10 mixed responsibilities. Easier to test, easier to find bugs.

Running Locally

Start infrastructure

docker-compose up -d

Run tests

JAVA_HOME=/usr/lib/jvm/java-21-openjdk-amd64 \
  mvn test -f backend/pom.xml

Start service

JAVA_HOME=/usr/lib/jvm/java-21-openjdk-amd64 \
  mvn spring-boot:run \
  -f backend/pom.xml \
  -pl url-shortener-service

Try it

# Shorten a URL
curl -X POST http://localhost:8080/api/v1/shorten \
  -H "Content-Type: application/json" \
  -d '{"originalUrl":"https://www.google.com"}'

# Response:
# { "shortCode": "1", "shortUrl": "http://localhost:8080/r/1" }

# Resolve (follow the redirect)
curl -L http://localhost:8080/r/1

# Swagger UI
open http://localhost:8080/swagger-ui.html
