Why Rate Limiting?

Without a rate limiter…

  • A buggy client loops → server OOM crash
  • A scraper hammers the API → data leak + cost explosion
  • One noisy tenant starves all others
  • DDoS attack takes down the entire platform

With a rate limiter…

  • Predictable load — capacity planning is possible
  • Fair usage across all clients
  • Protection for downstream services
  • Clear API contracts (quotas communicated upfront)

Algorithm 1 — Token Bucket ✓ Implemented

🪣
Token Bucket
capacity: N = 10 | refill: R = 2/sec
7 tokens remaining

How it works

  • Bucket holds max N tokens
  • Refilled at rate R tokens/second
  • Each request consumes 1 token
  • Bucket empty → reject with 429

Properties

  • Allows controlled bursts — full bucket = N instant requests
  • Average rate bounded by refill rate R
  • O(1) per request — one atomic Redis counter

Why Lua scripts in Redis?

GET then DECR = two commands. Between them another request slips through. Lua executes atomically — no race condition possible.

requests · 15s elapsed
effective = curr + prev × (1 − elapsed/window)
effective = 30 + 80 × (1 − 15/60)
effective = 30 + 60 = 90
90 < 100 limit → ALLOW ✓

Why this fixes Fixed Window

  • No hard boundary to exploit
  • Weight of previous window decreases as current window progresses
  • Approximation — but accurate enough for all practical purposes
  • O(1) — two Redis INCR + one formula

Fixed Window flaw (why we don't use it)

12:01:00 → 100 req [start of window B]

Data Model — Domain Layer

RateLimitKey

Value Object 

type: "user"
id:   "123"
→ value: "user:123"

  • Wraps a String — prevents passing "user:123" where "apikey:abc" is expected
  • Validates format on construction
  • Equality by value (record)

RateLimit

Value Object 

maxRequests: 100
window: Duration.ofSeconds(60)
algorithm: TOKEN_BUCKET

  • Duration not int — self-documenting, no unit ambiguity
  • Rejects maxRequests ≤ 0
  • Rejects null or zero window
  • Immutable rule — configured per client type

RateLimitResult

Value Object 

allowed:    true/false
remaining:  42
resetAt:    Instant
retryAfter: Duration

  • allowed=false → HTTP 429
  • retryAfter → Retry-After header
  • RFC 6585 compliance
  • Clients back off automatically

Why all three are Value Objects (Java records)? They're immutable snapshots — a rate limit rule doesn't change, a result doesn't change. Records give you equals/hashCode/toString for free, based on all fields.

Key Design Decisions

DecisionChoiceWhy
Counter storage Redis Distributed, atomic ops (INCR), TTL auto-expires old keys — no cron job needed
Atomicity Lua scripts GET + DECR must be a single atomic unit. Lua scripts run single-threaded in Redis
Two algorithms Token Bucket + Sliding Window Token Bucket = bursts OK (APIs). Sliding Window = accurate at boundaries (billing)
No database Redis only Rate limit state is ephemeral — losing it on restart is acceptable, unlike bookings
Response on limit HTTP 429 + Retry-After RFC 6585. Well-behaved clients back off automatically without hammering the service
Key format rl:{algo}:{type}:{id}:{epoch} Namespaced to avoid Redis key collisions. Epoch buckets auto-partition by window

AWS Equivalent Informational only — not implemented

In production you'd use managed services. We build it ourselves to understand what they do internally.

What we buildAWS managed serviceNotes
Token Bucket per API key API Gateway Usage Plans Burst limit + steady-state rate per API key. Managed, no code needed.
Sliding Window per IP AWS WAF Rate-Based Rules 5-minute sliding window per IP. Auto-blocks. Works at CloudFront edge.
Standalone limiter service Lambda Authorizer Runs before every API Gateway request. Can read DynamoDB counters.
Redis counters + Lua ElastiCache (Redis) Same data structures, managed cluster, auto-failover, Multi-AZ.
Custom algorithm selection Not available managed AWS WAF is fixed-window. Building your own = full algorithm control.

Interview tip: Always mention the trade-off between build vs buy. Managed services = less ops burden. Custom build = full control over algorithms, latency, and cost.
