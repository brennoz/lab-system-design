Web Crawler

BFS · Bloom Filter · Politeness Delay · URL Normalisation

RabbitMQ Redis PostgreSQL Jsoup Spring Boot 3.3 Java 21

Starts from seed URLs, fetches pages, extracts links, recursively crawls those links.

The web is a cyclic directed graph. Every design decision here is about graph traversal at scale.

What is a Web Crawler?

Program that browses the web automatically. Start at some pages, read them, find links, follow links, repeat forever.

Analogy

  • Get handed one book
  • Read it, note every book it references
  • Fetch all those books
  • Repeat for every new book found

Library = internet. Books = web pages. References = hyperlinks.

Seed URL

Starting point. The crawler has no idea where to begin until you give it one.

seeds = [
  "https://wikipedia.org",
  "https://bbc.com",
  "https://github.com"
]

From 3 seeds → hundreds → thousands → millions. The whole web is reachable.

Why it is hard

  • Web is a cyclic graph — a→b→a loops forever without deduplication
  • Crawling too fast = DDoS attack on target servers
  • Sites generate infinite URLs dynamically
  • Distributed workers need shared state

What we build

  • BFS traversal — correct graph algorithm
  • Bloom Filter — fast URL deduplication
  • Politeness delay — 1 req/sec per domain
  • URL normalisation — collapse duplicate URLs
  • RabbitMQ frontier — distributed work queue

Algorithm: BFS vs DFS

The web is a directed graph. BFS processes all neighbours before going deeper.

BFS — Breadth First ✓

Level 0: wikipedia.org
Level 1: history.com, bbc.com, github.com
Level 2: archives.gov, reuters.com, ...

→ Important pages found first
→ Popular = shallow = found early

DFS — Depth First ✗

wikipedia.org
  → history.com
    → archives.gov
      → deeper...
        → even deeper...
          (1000 obscure pages before touching bbc.com)

Risk: dive deep into a crawler trap before visiting important pages.

BFS pseudocode

frontier = Queue([seed_urls])
seen     = BloomFilter()

while frontier not empty:
  url = frontier.dequeue()

  if seen.mightContain(url):
    continue          // already visited

  seen.add(url)

  html = fetch(url)   // HTTP GET
  links = extract(html)

  for link in links:
    norm = normalise(link)
    if not seen.mightContain(norm):
      seen.add(norm)
      frontier.enqueue(norm)

  store(url, html)    // save to DB

Politeness Delay + Crawler Traps

Politeness Delay

Crawling bbc.com with 1000 concurrent threads = DDoS attack. Real crawlers enforce a per-domain delay.

// After fetching bbc.com/news:
SET crawl:lock:bbc.com "" EX 1
// key expires in 1 second

// Next bbc.com URL dequeued:
EXISTS crawl:lock:bbc.com → true
→ re-enqueue, skip for now

Redis TTL = automatic expiry. No cron job needed.

Crawler Traps

Sites that generate infinite unique URLs:

// Session ID trap
/news?session=abc123
/news?session=def456   ← same page, different URL

// Calendar trap
/calendar/2024/01
/calendar/2024/02
...
/calendar/9999/12      ← infinite

// Tracking params
/article?utm_source=twitter
/article?utm_source=facebook  ← same page

URL Normalisation fixes traps

  • Lowercase scheme + host
  • Strip URL fragment (#section)
  • Strip tracking params (utm_*, fbclid)
  • Max depth = 3 hops from seed

Why RabbitMQ, not Kafka?

FeatureKafkaRabbitMQ
Message replay✅ Yes❌ No
Message ordering✅ Per partitionApproximate
RetentionDays/weeksUntil consumed
ThroughputMillions/secThousands/sec
ComplexityHighLow
Use caseEvent sourcing, audit logWork queue

URL Frontier needs:

  • URL in → one worker picks it up → done → discard
  • Retry on worker crash (Dead Letter Queue)
  • No replay of "all URLs crawled last Tuesday"
  • No strict ordering needed

That is a Work Queue

RabbitMQ is purpose-built for exactly this. One URL processed by exactly one worker. ACK on success. NACK on failure → DLQ.

Analogy

Kafka = freight train. Great for large volumes with guaranteed delivery history.

RabbitMQ = delivery van. Gets the package there. Once delivered, it is gone.

Using Kafka here = freight train to deliver one pizza.

Use Kafka in this lab for:

  • Project 09 — Bank: Event Sourcing (must replay)
  • Project 10 — News Feed: stream processing
  • Project 11 — Twitter: audit log + fan-out

Hexagonal Architecture

Same pattern as project 05. Domain knows nothing about Spring, Redis, RabbitMQ, or HTTP.

domain/ — pure Java
  • CrawlUrl, Domain, CrawlPage — value objects
  • UrlNormaliser, LinkExtractor — domain services
  • 5 ports: SeenFilterPort, UrlQueuePort, PolitenessPort, PageFetcherPort, PageRepository
application/
  • SubmitSeedUseCase — accepts seeds from API
  • CrawlUrlUseCase — processes one URL from queue
infrastructure/
  • JpaPageRepository — PostgreSQL
  • RedisSeenFilter — Bloom Filter
  • RabbitMqUrlQueue — publish + @RabbitListener consumer
  • RedisPolitenessAdapter — SET EX / EXISTS
  • HttpPageFetcher — RestTemplate
api/
  • CrawlerController — POST /seeds, GET /status

New concept: @RabbitListener

In project 05, the use case was called directly by the controller. Here, CrawlUrlUseCase is also triggered by RabbitMQ messages — Spring calls it automatically when a URL arrives in the queue.

// Infrastructure adapter wires
// use case to RabbitMQ consumer
@RabbitListener(queues = "url.frontier")
public void onMessage(CrawlUrl crawlUrl) {
    crawlUrlUseCase.crawl(crawlUrl);
}

Use case itself has no RabbitMQ imports. Spring wiring in infrastructure only.

5 ports vs 2 in project 05

More ports = more moving parts = more opportunities to learn the pattern. Each port can be swapped independently — e.g. replace Redis Bloom Filter with Guava in-memory for tests.

Running Locally

Start infrastructure

docker-compose up -d

# RabbitMQ management UI:
open http://localhost:15672
# login: guest / guest

Run tests

JAVA_HOME=/usr/lib/jvm/java-21-openjdk-amd64 \
  mvn test -f backend/pom.xml

Start service

JAVA_HOME=/usr/lib/jvm/java-21-openjdk-amd64 \
  mvn spring-boot:run \
  -f backend/pom.xml \
  -pl web-crawler-service

Try it

# Submit seed URLs
curl -X POST http://localhost:8080/api/v1/seeds \
  -H "Content-Type: application/json" \
  -d '{"urls":["https://wikipedia.org"]}'

# Check crawl status
curl http://localhost:8080/api/v1/status

# Response:
# { "pagesCrawled": 42, "queueDepth": 318 }
